결제 페이지 라이브 채팅 도입
Authored by Gaurav Oberoi. Last updated on Sep 28, 2016.

이 프로젝트의 목표는 결제 페이지에 라이브 채팅을 도입하여 전환율을 높이는 것이다. 실험 기간은 30일이며, 그 이후에 이 기능을 유지할지, 아니면 걷어낼지 결정할 것이다.

프로젝트를 2줄이나 그 이하로 설명하세요. 여기서의 한 줄은 데스크탑에서 일반적으로 작성하는 문서 (e.g., Google Docs, wiki, text file)에서의 한 줄을 의미합니다. 가독성을 좋게 하기 위해서는 길이를 제한하는 것이 중요합니다.
Overview
Problem
현저히 낮은 결제 전환율: 결제 페이지에서 “예약" 버튼을 누르는 유저의 비율이 18% 밖에 되지 않는다. 유사한 호텔 서비스들의 경우 ~30% 의 수치를 가지고 있다(source). 유저를 놓치고 있다!
알 수 없는 이탈 원인: 기존 고객 문의 내역과 설문조사 결과에서 이탈 원인을 분석한 결과 어메니티 관련 문의, 가격, 체크아웃 시간 등, 자잘한 이슈들의 롱테일을 볼 수 있었다. 이탈 원인의 명확한 분류가 어려운 상황이다.
도움말 추가는 큰 도움이 되지 않음: 저번 쿼터에 도움말을 추가하고 UI를 개선한 버전을 A/B 테스트 해본 결과, 전환율 1.5% 증가에 그쳤다.

가독성을 위해 불릿 포인트를 사용할 것을 강력히 추천합니다. 강조를 위해 볼드체를 사용하고, 각각의 불릿은 2-3줄 이내가 되도록 하는게 좋습니다. 숫자로된 리스트를 사용하면 말로 설명해야 할 때 편합니다.
Goals
라이브 채팅이 수익성을 갖고 전환율을 높일 수 있는지 테스트: 라이브 채팅 운영 인력을 생각했을 때, 하루에 90 주문 이상이 추가되어야 break even이다. 가능할지는 확실히 모르겠다. 이건 테스트니까!
테스트를 싼 값에 준비: 과한 치장을 피한다. 테스트 결과가 성공적이면 그런 부분들은 Q1에 작업할 것이다.
12월 1일까지 결과 필요: Q1 resource planning 미팅까지 8주 남았다.

목표를 보고 “우리가 이걸 달성했나?” 라는 질문에 쉽게 답변할 수 있도록 목표를 세워야 합니다.
Out of Scope
UX의 큰 변화: 우리는 단지 채팅 버튼만을 추가할 것이다. 다른 UX 변화는 없다.
채팅 서비스 공급자 최종 결정: 일단 SnapEngage를 사용해서 테스트하고, 이후에 다시 검토한다.
다국어 지원: 테스트를 간단히 진행하기 위해 영어 사용자만 대상으로 한다.

범위를 넘어서는 가정을 줄이고, 기대 수준을 맞추기 위한 부분입니다.
People and Roles
Heather (운영팀 매니저): 운영 인력 이용에 대한 우리의 가정을 검토, 승인함.
Randy (라이브 채팅 담당자): 직접 라이브 채팅을 운영하고, 일주일에 한 번 결과 공유.
Colin (개발자): 개발 및 QA. SnapEngage 관련 설정을 담당하고, 실제 이용 방법을 공유.
Kalpana (재무팀): 테스트에 대한 재무적 가정을 검토하고, 이후의 결과에 대해서도 검토.
Joha (디자인): 채팅 버튼 삽입에 대한 간단한 리뷰 담당. 주요한 디자인 변경은 없음.
Vikram (BI): 이 테스트로 우리가 확인하고자 하는 결과 지표를 필요할 때 얻을 수 있는지 검토.

이 프로젝트에 관련된 구성원이 누구인지, 어떤 역할을 맡는지를 정리한 부분입니다. 의사 결정에 거부/동의 권한이 있거나, 직접적으로 프로젝트에 참여하는 사람만을 포함합니다.
Context
이 부분은 문제나 제안 사항에 대한 추가적인 설명이 필요할 때 사용합니다. 활용 사례, 페르소나, 지표, 경쟁사 분석, 설문 결과 등 필요한 부분을 인용합니다.
Use Cases
유저가 원하는 것
2분 안에 도움 받기: 달성 가능한 수치인지 모르겠으나, 일단 2분을 목표로 한다.
모바일이나 데스크탑에서 모두 이용: 결제의 28%는 모바일에서 진행된다. 따라서 모바일 지원이 꼭 필요하다.
운영팀이 원하는 것
라이브 채팅 업무 수행: 데스크탑/웹 클라이언트를 사용. 모바일은 불필요함.
담당자 추가/삭제: 개발팀 지원 없이 운영팀 내부에서 담당자 추가/삭제가 가능해야 함.
업무시간 설정: 라이브 채팅 버튼이 보이고/사라지는 시간을 조정 가능.
유저 정보 열람: user-id를 전달해, 채팅 담당자가 문의가 들어온 고객의 정보에 대해 확인할 수 있어야 함.
태그: 카테고리 분류를 위해 채팅 완료 후, 담당자가 텍스트 형태의 태그를 커멘트로 남길 수 있어야 함.
Assumptions
Break even을 위해서는 운영 인력 1명당 하루에 90 주문이 추가되어야 함: 인건비와 고객 lifetime value를 토대로 계산한 결과(스프레드시트 참고)
20% 트래픽에게 라이브 채팅 기능 노출: 대기 시간, 채팅 시간, 동시에 진행 가능한 채팅 수 등에 대한 가정을 세워 계산함. 아직 적절한 추측을 위한 데이터가 전혀 없기 때문에 우선 20%로 정하고 상황에 따라 조정이 가능하도록 하였음
전환율이 18%에서 20%으로 상승해야 함: 성공을 위해 필요한 전환율 수치(퍼널 레포트 참고)
Proposal
제안 사항을 당신에게 가장 자연스러운 방법으로 표현하세요. 섹션을 나누고, 서사적인 흐름으로 작성하시면 됩니다. 프로젝트에 따라 와이어프레임, 유저 플로우 다이어그램, form input/validation, 데이터 구조 등이 필요할 수도 있습니다.
Live Chat Vendor
우선 명시된 use-case들을 지원하고, 저렴한($60/month) 가격을 가진 SnapEngage를 골랐다. 이 기능이 성공적인 것으로 판단되면 더 적절한 서비스 제공자를 다시 선정할 것이다. 유료 계정을 만들었으며, 인증서를 올려놓았다.
User Experience
SnapEngage 가이드 문서 에서 어떤 식으로 작동하는지 확인할 수 있다.
버튼: “지금 채팅하기” 로 문구를 설정하여 상세 페이지의 “예약하기" CTA 버튼 근처에 위치시키면 된다.
반응: 버튼을 클릭하면 담당자의 이름을 보여주고 “어떻게 도와드릴까요?”라는 메시지를 보여준다.
모든 결제 페이지에서 노출: 3단계로 이루어진 장바구니-결제 페이지 모두에서 유지되어야 한다.
자동 팝업은 사용하지 않음: 나중에 시도해볼 수도 있겠으나, 지금은 하지 않는다.
Chat Variables
설명: 채팅 서비스 제공자의 javascript를 넣을 때 운영 담당자가 보게 되거나, 또는 레포팅에서 사용할 추가 변수를 전달한다.
항목: user_id, user_email, user_agent, booking_id, booking_order_value.
Knobs for Running the Test
페이지 트래픽의 일부에게만 이 기능을 노출할 것이다. 상세한 내용은 다음과 같다.
X% 유저에게만 노출: 코드 배포 없이 X를 조절할 수 있어야 한다. 운영팀 요청 시 DB/Redis 등을 이용해서 개발팀이 수동으로 업데이트 할 수만 있어도 된다.
채팅 버튼을 한번 본 유저는 계속 보도록: 만약 어떤 유저가 라이브 채팅 버튼을 한 번이라도 봤다면, 테스트를 진행하는 동안에는 계속 볼 수 있어야 한다. 쿠키 이용 또는 user_id를 이용(e.g. user id mod 100 <= X 인 유저에게 노출).
비율 늘리기: 운영에 문제가 없다면 첫 날에는 5%, 둘째 날에는 10%, 그리고 셋째 날 부터는 20%에게 노출한다.
Metrics and Reports

새로운 레포트:
라이브 채팅 기능 적용 대상 고객(실험 집단)과 그렇지 않은 고객(통제 집단) 각각의 전환율 
라이브 채팅 진행 후 발생한 주문 수와 매출
라이브 채팅 노출 고객 중 얼마나 많은 유저가 클릭해서 참여했는지
SnapEngage 레포트: 채팅 당 소요 시간, 동시에 진행된 채팅 수 등에 대한 레포트

위에 적은 예시가 너무 대략적일 수 있습니다. 이런 부분은 engineering과 BI 팀이 더 도와줄 수 있는 부분이고, 이게 바로 그들이 이 문서 검토에 참여해야 하는 이유입니다. 무엇이 필요한지 직접 실행할 사람들이 이해할 수 있게 적으면 됩니다.
Future Work
채팅 사용량이 너무 적었다면: 다음 아이디어를 시도 해볼 수 있다. (a) 채팅화면이 자동 팝업으로 뜨도록 한다, (b) CTA 카피나 UI를 변경해본다, (c) 채팅 CTA 옆에 담당자의 사진을 넣는다.
테스트가 성공적이였다면: 운영 담당자 리소스 추가 배정을 요청한다. 서비스 제공자를 선택하고, 레포팅을 강화하고, 담당 인력을 정식으로 트레이닝해서 Q1에 정식으로 기능을 출시한다.

이후의 상황을 미리 내다보고 준비할 수 있도록 적어주는 것도 좋습니다.
